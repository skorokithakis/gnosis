gnosis — a knowledge base for what the code can't tell you

PURPOSE

Code documents what a system does, documentation describes how to use
it, but there's a third kind of knowledge that neither captures: why
things are the way they are, what was tried and abandoned, what's known
to be broken, and what people intend to change but haven't yet.

gnosis records these intangibles so they survive context switches,
personnel changes, and the passage of time.

WHAT TO RECORD

  - A decision and the reasoning behind it.
  - An alternative that was considered and rejected.
  - Something known to be flawed or fragile.
  - An intent to change something that hasn't happened yet.
  - A constraint that shaped a design in a non-obvious way.
  - A lesson learned the hard way.

WHAT NOT TO RECORD

If someone reading the code or existing documentation would learn it,
don't put it here. Duplication rots. The value of gnosis is that it
holds what isn't written anywhere else.

Don't record:
  - How a function works (that's what the code is for).
  - API contracts or configuration (that's documentation).
  - TODOs that belong in a ticket tracker.
  - Anything that will be stale in a week — that's status, not
    knowledge.

THE AGENT LOOP

If you are an AI agent, gnosis has three touchpoints in your workflow:

1. After planning, before implementing:
   You've read the task and laid out a plan. Before writing any code,
   run "gnosis search" with keywords from your plan — system names,
   feature areas, technical concepts you're about to touch.

   Read what comes back. If any recorded decision conflicts with your
   plan, surface it to the user before proceeding. The recorded
   decision may still hold, may be stale, or may need explicit
   supersession — but don't silently contradict it.

2. During work:
   If a decision is made or an alternative is explicitly rejected,
   record it immediately with "gnosis write" rather than waiting for
   the end of the session. Context may be lost before you get to the
   review step.

3. After finishing work:
   Run "gnosis help review" and follow its guidance.

TIDYING

Stale knowledge is worse than no knowledge. When a recorded decision
is overturned, either remove the old entry ("gnosis rm") or write a
new one that references it ("gnosis write ... --related <old-id>")
explaining what changed. Don't leave contradictions in the system.

TOPICS

Entries are tagged with one or more topics — free-form labels like
"KeymasterTokenAuth" or "billing-reconciliation". Topics are:
  - Created on first use. No declaration needed.
  - Case and separator insensitive on lookup.
    "KeymasterTokenAuth", "keymaster-token-auth", and
    "keymaster_token_auth" all resolve to the same topic.
  - Displayed in normalized form (lowercase, dashes between words).

Use topics that match the system's shape: subsystem names, feature
areas, service names. They'll evolve as the codebase does.

COMMANDS

  gnosis help                Show this text.
  gnosis help review         Guidance for reviewing a session (read this
                             after finishing work, before writing entries).
  gnosis write <topics> <text> [--related id,id]
                             Record an entry. Topics are comma-separated.
  gnosis search <query>      Search entries by keyword.
  gnosis show <id|topic>     Show one entry or all entries for a topic.
  gnosis topics              List all topics with entry counts.
  gnosis edit <id>           Edit an entry in $EDITOR.
  gnosis rm <id> [id...]     Remove entries.
  gnosis reindex             Rebuild the search index.

Entries are identified by short IDs (six lowercase letters), printed
when you create them. Each entry tracks when it was created and last
updated.
